# Агенты и протокол оркестрации

## Роли
- Rules Agent — валидирует допустимость действий, определяет необходимость проверки, навык и DC
- Character Agent — вычисляет модификаторы персонажа (ability/skill/equipment/temporary)
- Narrative Agent — превращает исход механик в связный художественный текст
- GM Agent — агрегирует ответы, подводит итоги и выдаёт финальный ответ игроку

Детерминизм: решение об успехе/провале всегда принимает код механик, а не модель.

## Контракты (упрощённо)
```ts
// Вход чата (от игрока)
export interface PlayerInput {
  sessionId: string;
  content: string;
  profile?: PlayerProfile; // опционально: класс и краткая информация о персонаже
}

export interface PlayerProfile {
  name?: string;      // имя персонажа (по желанию)
  className?: string; // произвольный класс (свободная строка)
  bio?: string;       // краткая информация о себе
}

// Ответ Rules
export interface RulesOutput {
  requiresCheck: boolean;
  skill?: string; // например, "athletics"
  dc?: number;    // Difficulty Class
  reason?: string;
}

// Ответ Character
export interface CharacterOutput {
  modifiers: {
    ability: number;
    skill: number;
    equipment: number;
    temporary: number;
    total: number; // сумма остальных
  };
}

// Итог механики
export interface CheckOutcome {
  success: boolean;
  critical: boolean; // roll == 1 или 20
  margin: number;    // modified - dc
}

// Ответ Narrative
export interface NarrativeOutput {
  text: string; // художественное описание
}

// Итог GM
export interface GMOutput {
  text: string; // финальный ответ игроку
  summary?: string; // краткий итог хода (для журнала)
}
```

## Оркестрация
1) Параллельно запрашиваем `Rules` и `Character` (тайм-аут, отмена)
2) Если `requiresCheck = true`, выполняем бросок и расчёт исхода в коде
3) Передаём результат в `Narrative`, получаем текст
4) `GM` агрегирует: приоритет механик > нарратива; оформляет итог

## Политика при конфликте
- Истина — в детерминированных функциях механик; при расхождении текста и механики побеждает механика
- Отсутствующие части ответа агентов заполняются безопасными значениями (fallback)

## Временные ограничения
- Каждый агент исполняется с тайм-аутом (например, 5–8 секунд)
- Общий бюджет времени хода ограничен (например, 12–15 секунд)

## Политика бросков (ответственность Rules)
Rules Agent принимает решение «когда бросать кубы» согласно разделу «Когда бросать кубы» (docs/02-game-mechanics.md):

- Не бросаем, если действие тривиально (автоуспех), противоречит фикции (автопровал), чисто описательное, либо провал блокирует сюжет (успех с ценой).
- Бросаем, когда есть неопределённость и значимые ставки. Тип:
  - `skill` — статический DC, без активного противника;
  - `contest` — активное противодействие (NPC/среда);
  - `save` — спасбросок от внешнего эффекта.
- Rules возвращает: `requiresCheck`, `type`, `skill?`, `dc?`, `opposedBy?`, `rationale`, `alternative?`.

GM использует решение Rules, но итог всегда определяет детерминированная механика. В случае конфликта приоритет у кода механик над текстом моделей.

## Гибридная стратегия Rules (LLM‑классификатор)
- Язык промптов и ответов по умолчанию: RU
- Pipeline:
  1) Heuristic Gate (код): автоуспех/автопровал/нет ставок/успех с ценой
  2) LLM‑классификатор (минимальный JSON без текста) — определяет `type`, `skill`, `dc`, `rationale`, `alternative`
  3) Sanitizer (код): нормализует навык и DC, применяет дефолты и границы
- Контракт LLM‑ответа валидируется Zod‑схемой. Температура низкая, токены ограничены (например, 128–256). Few‑shot примеры намерений.
- При низкой уверенности `requires_clarification: true` и короткий `ask`; GM задаёт уточняющий вопрос и повторяет классификацию.

### Heuristic Gate (детали)
Реализация: `lib/agents/heuristics.ts`

**Логика автоматического решения:**
- Тривиальные действия без ставок → `auto_success` (не требуется проверка)
- Примеры тривиальных действий: осмотреться, поднять предмет, подойти к двери, улыбнуться
- Действия со ставками → передаются в LLM-классификатор
- Примеры ставок: опасность, стражники, враги, высота, темнота, ловушки, скрытность

**Нормализация текста:**
- Приведение к нижнему регистру
- Замена `ё` на `е` для единообразия
- Удаление лишних пробелов

**Тестовое покрытие:** `lib/agents/heuristics.test.ts`
- Тривиальные действия (осмотреться, поднять меч, подойти к двери, улыбнуться)
- Действия со ставками (темная комната с ловушками, стражники, обрыв, скрытность)
- Граничные случаи (пустой ввод, регистронезависимость, нормализация `ё`)
