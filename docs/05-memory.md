# Память: MVP и развитие

## MVP: контекст на основе последних сообщений
- Используем последние 12–15 сообщений только двух ролей: Игрок (Player) и GM
- Источники сообщений вспомогательных агентов (Rules/Character/Narrative) в контекст НЕ попадают
- Храним сообщения в БД и при сборке промпта выбираем по `sessionId`, ролям и дате

Пример выборки (последние N):
```sql
SELECT role, content
FROM "Message"
WHERE session_id = $1
  AND role IN ('player','gm')
ORDER BY created_at DESC
LIMIT 15;
```

## Схема данных (предложение)
- `Session(id, created_at, ... )` — сессия игры
- `Message(id, session_id, role, content, created_at)` — чатовые сообщения
- `Turn(id, session_id, player_input_id, gm_output_id, meta, created_at)` — агрегированный ход (ссылки на сообщения + метаданные)
- `Roll(id, session_id, value, modified, category, modifiers_json, created_at)` — броски и вычисления

Такой дизайн даёт:
- Простое построение контекста (Message по ролям)
- Воспроизводимость (Turn связывает вход и выход; Roll фиксирует числа)
- Отделение детерминированной логики от нарратива

## Политики
- Запись: 1 сообщение игрока → 1 итоговое сообщение GM; вспомогательные — не сохраняем как отдельные сообщения
- Очистка: по необходимости добавим TTL/архивацию старых сессий
- Приватность: хранить только необходимое; персональные данные не писать

## Дальнейшее развитие: векторная память
- При необходимости добавим таблицу `Memory` с колонкой `embedding vector` и HNSW/IVFFLAT индексом
- Записывать только важные (IMPORTANT/QUEST) события; ограничение на число записей за ход; дедупликация по близости
- Запросы памяти — через косинусную близость к текущему запросу; суммаризация перед подачей в модель
