# Игровые механики

## 📚 Обзор

Документ описывает реализованные и планируемые игровые механики RPGate. Все механики строятся по принципам:
- Детерминированная логика (чистый код) отделена от генеративной (LLM)
- Расширяемость через модульные плагины (`mechanics/`)
- Минимум скрытого состояния: всё важное сохраняется в БД
- Feature flags управляют включением новых систем

## 🎲 Система бросков (Dice System)

### D20 Базовый бросок
Используется для большинства проверок навыков.

| Roll | Категория       | Интерпретация | Доп. эффект |
|------|-----------------|---------------|-------------|
| 1    | Critical Failure| Усиленный провал | Побочный негативный эффект |
| 2-9  | Failure         | Провал        | — |
| 10-19| Success         | Успех         | — |
| 20   | Critical Success| Усиленный успех | Бонус, дополнительная деталь |

```typescript
export interface DiceResult {
  roll: number;
  modified: number;
  category: 'CRIT_FAIL' | 'FAIL' | 'SUCCESS' | 'CRIT_SUCCESS';
  modifierBreakdown: {
    ability: number;
    skill: number;
    equipment: number;
    temporary: number;
    total: number;
  };
}

export function classifyD20(roll: number) {
  if (roll === 1) return 'CRIT_FAIL';
  if (roll === 20) return 'CRIT_SUCCESS';
  return roll < 10 ? 'FAIL' : 'SUCCESS';
}
```

### Модификаторы
Итог: `modified = roll + totalModifier`. Источник модификаторов:
- ability (стат характеристик)
- skill (прокачка навыков)
- equipment (экипировка / предметы)
- temporary (баффы / дебаффы)

### Преимущество / Помеха (Advantage / Disadvantage)
Планируется на будущие версии.
- Преимущество: 2 броска, берём больший
- Помеха: 2 броска, берём меньший

```typescript
function rollWithAdvantage(modifier: number = 0): DiceResult {
```

## 🧪 Проверки навыков (Skill Checks)

### Процесс
1. Rules Agent определяет: `requiresCheck` + `skillType` + `difficultyClass (DC)`
2. Character Agent предоставляет модификаторы на основе состояния персонажа (в т.ч. выбранного класса и характеристик)
3. Пользователь бросает кубик
4. GM интерпретирует исход, формирует нарратив

### DC Классификация
| DC   | Сложность       | Описание |
|------|-----------------|----------|
| 5-9  | Очень легко     | Почти невозможно провалить |
| 10-14| Легко           | Базовый челлендж |
| 15-19| Средне          | Необходима подготовка |
| 20-24| Сложно          | Требует хороших характеристик |
| 25-29| Очень сложно    | Эпизодические задачи |
| 30+  | Геройское       | Почти легендарно |

### Интерпретация итогов
```typescript
interface CheckOutcome {
  success: boolean;
  critical: boolean;
  margin: number; // modified - DC
}
```
- `margin >> 0` можно использовать для градации усиленных успехов
- При `CRIT_FAIL` игнорируем margin, применяем негативный эффект

## ⚔️ Боевая система (Combat) — MVP план

Пока заглушка. Первые шаги:
- Структура боя: раунды и инициатива
- Типы действий: атака, защита, использование предмета
- Формула урона: оружие + модификатор характеристики (+ критический бонус)

```typescript
interface CombatTurn {
  actorId: string;
  initiative: number;
  action: 'attack' | 'defend' | 'use_item';
  targetId?: string;
}
```

Будущее расширение:
- Статусы (оглушён, кровотечение)
- Позиционирование и дальность
- Многократные атаки (high-level персонажи)

## 🧠 Память и pgvector

### Типы памяти
| Тип         | Назначение                     |
|-------------|--------------------------------|
| IMPORTANT   | Критические сюжетные события   |
| RELATIONSHIP| Отношения с NPC / компаньонами |
| QUEST       | Шаги выполнения квестов        |
| LOCATION    | Описания посещённых мест       |
| ITEM        | Данные о важных предметах      |

### Поток сохранения памяти
1. GM помечает важное событие
2. Сервис памяти генерирует эмбеддинг
3. Сохраняет в таблицу `Memory` (vector + content)
4. Запросы: поиск релевантных воспоминаний по косинусной близости

```sql
SELECT id, content
FROM "Memory"
ORDER BY embedding <=> $1
LIMIT 5;
```

## 🔐 Консистентность механик

## 🔄 Расширение механик

## 🧭 Когда бросать кубы (Decision Policy)

Бросок нужен не всегда. Следуем принципам:

- Бросаем, если одновременно выполняются условия:
  - есть неопределённость исхода (не очевидно, получится ли);
  - есть значимые ставки (успех/провал меняют ситуацию, есть риск/бонус);
  - исход зависит от навыков/характеристик персонажа или внешних факторов;
  - провал не приводит к «мёртвой точке» сюжета без альтернатив.
- Не бросаем (нарратив/авто‑решение), если:
  - действие тривиально для персонажа с текущими параметрами (автоуспех);
  - действие противоречит фикции/логике мира (автопровал с объяснением);
  - это чисто описательное действие без ставок и рисков;
  - провал заблокировал бы прогресс: применяем «успех с ценой» вместо чека.
- Один чек на «намерение», а не на каждый микрошаг. Если задача многошаговая — используем прогресс (clock), а не серию бросков «на каждый шаг».
- Противодействие: если есть активное сопротивление NPC/среды — проводим состязание (contest): обе стороны делают бросок, сравниваем modified.
- Пассивные значения: для распознавания очевидного используем пассивные показатели (например, «пассивное восприятие» = 10 + модификаторы).

### Определение DC и модификаторов
- DC по базовой шкале (см. таблицу ниже) и корректируется обстоятельствами: освещение, инструменты, время, помощь (обычно в диапазоне −5 … +5 к DC или к итоговым модификаторам).
- Критические значения d20 (1/20) всегда учитываются: 1 — усиленный провал, 20 — усиленный успех.

### Выбор типа проверки
- Навык (skill check) с фиксированным DC — для задач без активного противника.
- Состязание (contest) — когда противодействие активно (скрытность vs. внимательность и т.п.).
- Спасбросок (saving throw) — когда персонаж избегает внешнего эффекта/опасности.

### Соответствие «намерений» навыкам (примеры)
- Прыжок/лазание/плавание — Athletics
- Скрытность/прятаться — Stealth
- Замок/ловушка/тонкая моторика — Sleight of Hand (и/или инструменты)
- Поиск улик — Investigation; замечать — Perception
- Выживание/следопыт — Survival
- Переговоры — Persuasion/Deception/Intimidation по контексту
- Знание лора — History/Arcana/Nature/Religion по контенту

### Алгоритм принятия решения (псевдокод)
```typescript
type CheckType = 'none' | 'skill' | 'contest' | 'save';

interface DecideCheckInput {
  intent: string;          // что хочет сделать игрок
  context: any;            // состояние сцены/мира
  character: any;          // ключевые показатели персонажа
}

interface DecideCheckResult {
  requiresCheck: boolean;
  type: CheckType;
  skill?: string;
  dc?: number;
  opposedBy?: string;      // если contest, кем/чем
  rationale: string;       // объяснение решения
  alternative?: 'auto_success' | 'auto_fail' | 'success_with_cost';
}

export function decideCheck(i: DecideCheckInput): DecideCheckResult {
  if (violatesFiction(i)) {
    return { requiresCheck: false, type: 'none', rationale: 'Противоречит ситуации/миру', alternative: 'auto_fail' };
  }
  if (trivialGivenStats(i)) {
    return { requiresCheck: false, type: 'none', rationale: 'Тривиально для персонажа', alternative: 'auto_success' };
  }
  if (!hasMeaningfulStakes(i)) {
    return { requiresCheck: false, type: 'none', rationale: 'Нет ставок, чистый нарратив' };
  }
  if (wouldBlockProgressOnFail(i)) {
    return { requiresCheck: false, type: 'none', rationale: 'Провал остановит сюжет', alternative: 'success_with_cost' };
  }
  if (hasActiveOpposition(i)) {
    return { requiresCheck: true, type: 'contest', opposedBy: 'npc/environment', rationale: 'Есть активное противодействие' };
  }
  const skill = inferSkill(i.intent, i.context);
  const dc = estimateDC(i.intent, i.context);
  return { requiresCheck: true, type: 'skill', skill, dc, rationale: 'Неопределённость и ставки присутствуют' };
}
```

### Примеры
- «Открыть обычную дверь» — автоуспех.
- «Перескочить узкий ров (1 м), атлетичный персонаж» — автоуспех; широкий ров (4–5 м) — проверка Athletics.
- «Попросить воды у трактирщика» — без чека; «выпросить секреты у стражи» — проверка Persuasion/Deception.
- «Спрятаться от патруля» — contest: Stealth (игрок) vs. Perception (сторожевые).
- «Взлом замка под дождём и спешкой» — skill + повышенный DC из‑за условий.

Шаги добавления новой механики:
1. Создать интерфейс / типы (`mechanics/<name>.ts`)
2. Реализовать детерминированную логику (расчёты, проверки)
3. Добавить интеграцию в соответствующий агент или отдельный новый
4. Включить feature flag
5. Добавить тесты (unit + интеграция)
6. Обновить документацию

## 🧪 Тестирование механик (кратко)
- Unit: формулы модификаторов, классификация броска (`lib/mechanics/dice.test.ts`)
- Unit: эвристические решения Rules Agent (`lib/agents/heuristics.test.ts`)
  - Тривиальные действия без ставок (автоуспех)
  - Действия со ставками (передача в LLM)
  - Нормализация текста и граничные случаи
- Contract: структура результатов проверки
- Integration: полный поток skill check
- Snapshot: GM интерпретации критических исходов

## 📏 Ограничения MVP
- Нет позиционирования в бою
- Нет системы усталости
- Магия, крафтинг, компаньоны — отключены
- Один персонаж на игрока
